7. Uruchomienie środowiska lokalnie
Sieć wewnętrzna Dockera
docker network create $NETWORK

Start 2x backendu — bez publikacji portu na hosta
api-a
docker run -d --name api-a --network $NETWORK -e INSTANCE_ID=api-a "${BACK_IMAGE}:${VERSION}"
api-b
docker run -d --name api-b --network $NETWORK -e INSTANCE_ID=api-b "${BACK_IMAGE}:${VERSION}"


Start frontendu — jedyny publiczny kontener
docker run -d --name frontend --network $NETWORK -p 8080:80 "${FRONT_IMAGE}:${VERSION}"

Aplikacja będzie dostępna pod:

http://localhost:8080/
----------------------------------------------------
Usuwanie:
docker rm -f frontend api-a api-b
docker network rm $NETWORK

Najważniejszy fragment (interpretacja pod zadanie)
COPY package.json package-lock.json ./   ← najpierw dependencies
RUN npm ci --omit=dev                   ← instalacja (cache!)
COPY . .                                ← kod aplikacji później

Polecenie `docker history` pokazuje kolejność warstw obrazu od najnowszej do najstarszej.

Górne warstwy odpowiadają instrukcjom z Dockerfile:
- CMD uruchamia aplikację przez npm
- EXPOSE definiuje port 3000
- USER node zapewnia uruchomienie jako użytkownik nieuprzywilejowany
- COPY . . kopiuje kod aplikacji
- RUN npm ci --omit=dev instaluje zależności produkcyjne
- COPY package.json package-lock.json umożliwia cache warstwy zależności

Kolejność warstw potwierdza optymalizację:
najpierw kopiowane są pliki zależności, następnie wykonywana jest instalacja, a dopiero później kopiowany jest kod aplikacji.

Dolne warstwy pochodzą z obrazu bazowego node:20-alpine i zawierają system Alpine oraz runtime Node.js.
